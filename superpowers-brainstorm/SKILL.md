## 🎯 When to Use This Skill

**ALWAYS use this skill** when:
- Starting ANY new task or project
- User asks questions that require planning
- Complex problems need breaking down
- User requests feature implementation
- Ambiguous requirements need clarification
- Multiple approaches are possible

**THIS IS A MANDATORY FIRST STEP** before any implementation work begins.

## 🧠 Brainstorming Framework

### Phase 1: Question Generation

**ALWAYS generate these questions before planning:**

#### Understanding Questions
1. What is the primary goal or objective?
2. What problem are we trying to solve?
3. Who are the users/stakeholders?
4. What are the success criteria?
5. What are the constraints (time, resources, technical)?

#### Context Questions
1. What is the current state/situation?
2. What has been tried before?
3. Why are we doing this now?
4. What is the broader context?
5. What are the dependencies?

#### Technical Questions
1. What tech stack are we using?
2. What existing patterns should we follow?
3. What are the technical constraints?
4. What integrations are needed?
5. What are the performance requirements?

#### Risk Questions
1. What could go wrong?
2. What are the biggest risks?
3. What are the unknowns?
4. What assumptions are we making?
5. What is our contingency plan?

## 📋 Planning Framework

### Phase 2: Plan Creation

**ALWAYS create a structured plan with these sections:**

#### 1. Executive Summary
- **Objective**: Clear, concise goal statement
- **Scope**: What's included and excluded
- **Timeline**: Estimated duration
- **Resources**: What's needed

#### 2. Current State Analysis
- **Situation**: What exists now
- **Gap Analysis**: What's missing
- **Technical Debt**: What needs addressing
- **Quick Wins**: Easy improvements

#### 3. Proposed Solution
- **Architecture**: High-level design
- **Approach**: Methodology
- **Technology Stack**: Tools and frameworks
- **Data Flow**: How data moves through the system

#### 4. Implementation Plan
- **Phase 1**: Foundation (must-haves)
- **Phase 2**: Core features (should-haves)
- **Phase 3**: Enhancements (nice-to-haves)
- **Phase 4**: Optimization (performance, polish)

#### 5. Risk Assessment
- **Technical Risks**: Implementation challenges
- **Business Risks**: Impact on operations
- **Mitigation Strategies**: How to address risks
- **Contingency Plans**: Backup options

#### 6. Success Criteria
- **Metrics**: How we measure success
- **Milestones**: Key deliverables
- **Validation**: How we verify it works
- **User Acceptance**: How users confirm value

#### 7. Open Questions
- **Unknowns**: What we need to find out
- **Decisions Needed**: What requires user input
- **Dependencies**: What we're waiting on
- **Assumptions**: What we're assuming

## 🔄 Interactive Brainstorming Process

### Step 1: Initial Question Round

```markdown
**I need to understand your requirements better. Let me ask some questions:**

**Understanding:**
1. What is the main goal you want to achieve?
2. What problem will this solve?

**Context:**
3. What is the current situation?
4. Why is this important now?

**Technical:**
5. What tech stack should I use?
6. Are there existing patterns I should follow?

**Constraints:**
7. Any time constraints?
8. Resource limitations?

Please answer these questions so I can create the best plan for you.
```

### Step 2: Analyze Responses

**ALWAYS:**
- Parse user's answers carefully
- Identify gaps in understanding
- Note constraints and preferences
- Detect any contradictions

## 💡 Brainstorming Techniques

### Technique 1: First Principles Thinking

```markdown
**Breaking down to fundamentals:**

1. What are we absolutely certain about?
2. What are the basic truths?
3. What can we infer from first principles?
4. What solutions emerge from these fundamentals?

**Questions to ask:**
- Why is this necessary?
- What would happen if we didn't do this?
- What's the simplest possible solution?
- What are we overcomplicating?
```

### Technique 2: Lateral Thinking

```markdown
**Approaching from different angles:**

1. How would [different domain] solve this?
2. What if we inverted the problem?
3. What's the opposite approach?
4. What metaphors apply here?

**Example approaches:**
- "How would a architect design this?"
- "What if we made it 10x simpler?"
- "What's the mobile-first approach?"
- "How would Netflix/Airbnb/Uber do this?"
```

### Technique 3: The 5 Whys

```markdown
**Getting to root cause:**

1. Why do we need this?
   → Because we need to [X]

2. Why do we need [X]?
   → Because [Y] is a problem

3. Why is [Y] a problem?
   → Because [Z]

4. Why does [Z] happen?
   → Because [root cause]

5. Why does [root cause] exist?
   → [fundamental reason]

**Solution emerges from understanding root cause.**
```

### Technique 4: Scenario Planning

```markdown
**Exploring different futures:**

**Best Case Scenario:**
- What if everything goes perfectly?
- What would that look like?
- How do we maximize this outcome?

**Worst Case Scenario:**
- What could go terribly wrong?
- How do we prevent this?
- What's our backup plan?

**Likely Scenario:**
- What's the most realistic outcome?
- How do we prepare for this?

**Wildcard Scenarios:**
- What unexpected events could occur?
- How do we stay adaptable?
```

## 🎯 Question Templates

### Template 1: New Feature Request

```markdown
**I understand you want to build [feature]. Let me ask some clarifying questions:**

**User & Context:**
1. Who will use this feature?
2. What problem does it solve for them?
3. When/where will they use it?
4. What's their current workaround?

**Functionality:**
5. What are the must-have features?
6. What are nice-to-have features?
7. Are there similar features we should reference?
8. What should this feature NOT do?

**Technical:**
9. Any technical constraints?
10. Integration with existing systems?
11. Performance requirements?
12. Data privacy/security considerations?

**Success:**
13. How will we know this is successful?
14. What's the primary metric to improve?

**Timeline:**
15. When do you need this?
16. Any hard deadlines?

This will help me create the perfect plan for you!
```

### Template 2: Bug Fix Request

```markdown
**I understand there's a bug. Let me gather details:**

**Bug Description:**
1. What exactly is happening?
2. What should happen instead?
3. When did this start happening?
4. How often does it occur?

**Reproduction:**
5. What steps reproduce this bug?
6. Is it consistent or intermittent?
7. What browser/device/environment?
8. Any error messages or logs?

**Impact:**
9. How many users are affected?
10. Is there a workaround?
11. What's the business impact?
12. Priority level?

**Context:**
13. What changed recently?
14. Related features/systems?
15. Similar bugs in the past?

This will help me diagnose and fix it properly!
```

### Template 3: Architecture Decision

```markdown
**You're asking about [technical decision]. Let me understand:**

**Current State:**
1. What's the current architecture?
2. What problems are you facing?
3. Why change now?

**Requirements:**
4. What are your technical requirements?
5. What are the constraints?
6. What are you optimizing for (speed/cost/maintainability)?

**Options:**
7. What options have you considered?
8. What are your initial thoughts?
9. Any preferences or red lines?

**Future:**
10. How do you see this evolving?
11. Any upcoming changes to consider?

This will help me recommend the best architecture!
```
